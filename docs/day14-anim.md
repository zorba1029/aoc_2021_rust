# Day 14: Extended Polymerization — 개수 시각화 (Count Visualization)

Day 14의 폴리머는 한 스텝마다 길이가 거의 두 배가 된다. 40스텝이면 ~20조 자라서
**문자열을 만들면 안 된다.** `day_14_6th.rs`처럼 **인접 쌍(pair)의 개수**만 추적한다.
이 문서는 그 개수가 폭발하는 과정을 터미널에서 **막대그래프 애니메이션**으로 보는
기록이다.

---

## 왜 이게 흥미로운가 — 지수 폭발과 "쌍으로 생각하기"

실제 문자열 대신 쌍을 세는 발상의 전환이 이 퍼즐의 핵심이다.

- `NNCB` → `{NN:1, NC:1, CB:1}`
- 규칙 `NN -> C` 는 쌍 `NN` 하나를 `NC`, `CN` 두 쌍으로 바꾼다. 개수는 그대로 옮겨간다:

  ```
  (a,b): count   ──rule(a,b)=r──▶   (a,r) += count,  (r,b) += count
  ```

이러면 상태 크기는 쌍 종류 수(≤ 규칙 수)로 **상수에 가깝게 유지**되는데, 표현하는
문자열 길이는 매 스텝 ~2배로 **지수 증가**한다. 막대는 매 스텝 정규화되어 모양은
읽을 수 있게 유지되지만, 옆에 찍히는 숫자가 천 → 백만 → 조 단위로 치솟는 걸 보면
지수 증가가 体感된다.

**글자 세기 트릭**: 각 글자는 인접 쌍에서 앞 글자로 한 번, 뒤 글자로 한 번 나온다.
그래서 "각 쌍의 **뒤 글자**만 세고 + 템플릿의 **맨 앞 글자** 1개"를 더하면 각 글자가
정확히 한 번씩 세어진다 (맨 앞 글자는 삽입으로 절대 안 바뀐다).

축약 엔진은 `day_14_6th.rs`의 쌍-개수 알고리즘을 쓰되, 한 스텝을 **빈 맵에서 새로
쌓기**로 구현했다 (clone/뺄셈 없이, 0 항목도 안 생김). 규칙 없는 쌍은 그대로 유지한다.

## 실행 방법

실제 터미널(VS Code 통합 터미널 OK, truecolor 지원)에서 실행한다.

```bash
# sample-a, 40스텝 — 뷰는 실행 시 프롬프트로 선택 (기본값)
cargo run --release --bin day14_anim

# 진짜 퍼즐 입력, View A(다단 막대)
cargo run --release --bin day14_anim -- input 40 a

# 진짜 퍼즐 입력, View B(히트맵 행렬)
cargo run --release --bin day14_anim -- input 40 b
```

인자:

- `arg1` — 입력 선택: `sample-a` | `input` (또는 raw 경로). 기본값 `sample-a`
- `arg2` — 최대 스텝. 기본값 `40`
- `arg3` — 쌍(pair) 뷰: `a`(=columns) | `b`(=matrix). **생략하면 실행 시 `[a/b]`
  프롬프트**로 선택 (A/B 테스트용)

### 정답 대조 (검증됨)

쌍-개수 로직은 AoC 정답으로 단위 테스트했다 (`cargo test --lib day14::animate`):

## 무엇을 보게 되나

### 1. 입력 로딩 단계

스텝 애니메이션 전에 입력을 한 줄씩 읽는 것처럼 보여준다 — **현재 읽는 규칙 하나만**
`i/전체` 카운터와 함께 (규칙 100개인 실제 입력도 자연스럽게 흘러간다):

```
  Day 14 — reading input

  template :  N N C B

  input   8/16 :  N N -> C        ← 맨 윗줄, 현재 입력만 + i/전체 카운터
```

### 2. 스텝 단계

쌍(pair) 표시는 **모든 쌍을 다 보여주는 두 가지 뷰**가 있다 (A/B 선택). 원소 막대와
답은 두 뷰 공통이다.

**View A — 다단 막대** (`a`): 모든 쌍을 개수 내림차순으로 4열 배치, 약식 숫자(1.2M).
```
  ── pairs ── (all 71, by count)
  VB ██████     931    VS ███        416    CO █          199    KO █           95
  BC ██████     918    BO ██         384    NN █          176    PH █           89
  ...
```

**View B — 히트맵 행렬** (`b`): `행=1번째 글자 × 열=2번째 글자`. 칸이 밝을수록 큰
개수(로그 스케일), 한 번도 안 나온 쌍은 `·`. 입력이 커도 모든 쌍이 한눈에.
```
  ── pairs ── (matrix: row=1st · col=2nd · brighter=more)
      B C F H K N O P S V
   B  █ █ █ █ █ · █ █ █ █
   C  █ █ █ █ █ · █ █ █ █
   ...
  scale (log): 0 → 931
```

공통 — **원소 막대 + 답**:
```
  ── elements ──
  B   ████████████████████████████████████████  3,585  ◀ MAX
  ...
  N   ███████                                      597  ◀ min

  answer (MAX - min): 2,988
```

- **쌍**: 이제 **상위 N개로 자르지 않고 전부** 보여준다(View A는 4열, View B는 행렬).
  첫 글자(원소)별 고정 색. 이게 `day_14_6th`의 `HashMap<(char,char),usize>` 자체다.
- **원소 막대**: 전체 표시, 원소별 고정 색, **MAX 굵게 + ◀MAX**, **min ◀min**.
- 막대는 각 섹션의 그 스텝 최댓값 기준 **선형 정규화**, 옆에 숫자(쌍은 약식, 원소는
  천 단위 콤마).
- 헤더의 `length`는 현재 폴리머 길이(원소 개수 합).

---

## 동작 원리 (구현 메모)

- **쌍 개수**: `HashMap<(char,char), u64>`. step40에서 최대 ~2.2조라 `u64`면 충분.
- **한 스텝**: `step()`이 빈 맵에서 `(a,r) += c`, `(r,b) += c`로 새로 쌓는다.
- **원소 개수**: `element_counts()` — 뒤 글자 합 + 맨 앞 글자 1.
- **막대 정규화**: 섹션별 최댓값 대비 `BAR_W`(기본 40칸) 비율, 0이 아니면 최소 1칸.
- **색**: 등장 원소를 알파벳순으로 8색 팔레트에 매핑(`build_colors`).
- **ANSI 렌더링** (day11/day18과 동일):
  - 대체 화면 진입/복귀 `\x1b[?1049h` / `l`, 커서 숨김/복원 `\x1b[?25l` / `h`
  - 매 프레임 `\x1b[H\x1b[J`(홈+끝까지 지움)로 제자리 갱신
  - 24비트 트루컬러 `\x1b[38;2;{r};{g};{b}m`
